# 当前代码中的时变信道设置说明

## 1. 结论先行

当前仓库里的“时变信道”已经统一成了 **sequence-level 状态机**，不是旧版那种“每个样本各自独立随机一次”的散落写法。

- 一条 `sequence` 只在开始时初始化一次信道状态。
- 同一条 `sequence` 内，每个 `frame` 通过递推更新信道。
- 递推的核心量是：
  - 路径时延 `ell`
  - 路径多普勒/时间尺度参数 `alpha`
  - 路径复增益 `h`
- 当前 paper 相关的 `vdop` / `vdop_ctrl` 入口，实际都默认使用 `doppler_mode = 'common_with_path_residual'`。

统一实现集中在以下 3 个文件：

- [`matlab/common/get_timevary_defaults.m`](../matlab/common/get_timevary_defaults.m)
- [`matlab/common/init_timevary_channel_state.m`](../matlab/common/init_timevary_channel_state.m)
- [`matlab/common/step_timevary_channel_state.m`](../matlab/common/step_timevary_channel_state.m)

## A. 按你关心的 1-5 直接回答

### 1. `alpha` 是怎么生成的

当前代码里，`alpha` 不是“每条路径一个永远固定的常数”，也不是“逐 sample 直接重抽”。它的粒度是：

- 一条 `sequence` 初始化一次状态；
- 一条 `sequence` 内，`alpha` 按 `frame` 递推；
- 每个 `frame` 内，`alpha_t` 是一个长度为 `P` 的路径向量，帧内不再二次更新。

具体支持 3 种生成方式：

1. `independent_path_ar1`
   - 每条路径各自做 AR(1)：
   - `alpha_t = rho_alpha * alpha_{t-1} + sqrt(1-rho_alpha^2) * alpha_max * randn`
   - 然后裁剪到 `[-alpha_max, alpha_max]`
2. `common_only`
   - 先递推一个公共运动状态 `v_norm_t`
   - 再令 `alpha_com_t = alpha_max * v_norm_t`
   - 各路径 `alpha_t = beta_path .* alpha_com_t`
   - 没有路径残差
3. `common_with_path_residual`
   - 先递推公共运动项 `alpha_com_t = alpha_max * v_norm_t`
   - 再递推路径残差
   - `delta_alpha_t = rho_delta * delta_alpha_{t-1} + sigma_delta * alpha_max * randn`
   - 最终 `alpha_t = beta_path .* alpha_com_t + delta_alpha_t`

当前 paper 主入口 `vdop` / `vdop_ctrl` 用的都是第 3 种，即：

- 有公共运动项；
- 有按路径的 residual Doppler；
- `alpha` 随 `frame` 慢变。

### 2. 时变是加在什么层面

当前实现不是“只在 DAFT/DAF 域加一个相位项”，也不是“只改一个统一 Doppler shift”。它是先在离散时域信道算子里写入 time-scaling，再做 AFDM 解调。

核心代码在 `build_timescaling_G_sparse(...)`：

- 采样位置：
  - `idx = (1 + alpha(i)) * n - ell(i) + Lcpp`
- 相位项：
  - `phase = exp(1j * 2 * pi * (alpha(i) * fc) * (n * dt))`

这表示：

- `ell(i)` 不是唯一的有效时延来源；
- 对固定路径 `i`，有效采样位置还会随 `n` 变化；
- 因而它更接近文献里的宽带 time-scaling 思路，也就是“等效时延随时间变化”；
- 然后这套离散时域算子再被映射成 AFDM 的等效 `Heff`。

如果只做二分判断：

- 它更接近“宽带 time-scaling 的离散实现”
- 不是“窄带统一多普勒频移 + 纯频域相位修正”

### 3. `alpha_max` 在代码里出现在哪些地方

`alpha_max` 当前会同时影响信道建模和后续 `c1` 设计，不只是一个“备注参数”。

#### 3.1 信道生成

- 在 [`matlab/common/get_timevary_defaults.m`](../matlab/common/get_timevary_defaults.m) 中作为默认上限给出
- 在 [`matlab/common/init_timevary_channel_state.m`](../matlab/common/init_timevary_channel_state.m) 中参与初始 `alpha` 生成
- 在 [`matlab/common/step_timevary_channel_state.m`](../matlab/common/step_timevary_channel_state.m) 中参与 AR/残差递推与裁剪
- 在 `common_*` 模式下，公共项满足 `alpha_com_t = alpha_max * v_norm_t`

#### 3.2 `c1` 计算

在下面两个主入口里，`c1_base` 都直接依赖 `alpha_max`：

- [`matlab/simulation/generate_dataset_timescaling_n256.m`](../matlab/simulation/generate_dataset_timescaling_n256.m)
- [`matlab/detectors/export_oracle_dataset_for_policy.m`](../matlab/detectors/export_oracle_dataset_for_policy.m)

具体依赖链是：

- `kmax = ceil((alpha_max * fc) * T)`
- `den = 1 - 4 * alpha_max * (N - 1)`
- `c1_base = (2 * kmax + 2 * alpha_max * (N - 1) + 2 * Nv + 1) / (2 * N * den)`

在线评估 [`matlab/detectors/run_online_policy_oamp_oampnet.m`](../matlab/detectors/run_online_policy_oamp_oampnet.m) 里也是同一口径。

#### 3.3 oracle sweep / RL action grid

`alpha_max` 不会直接出现在 action id 里，但它会先决定 `c1_base`，再决定 action grid：

- `ratios = linspace(0.6, 1.4, 21)`
- `c1_grid = c1_base * ratios`

因此：

- oracle sweep 遍历的是由 `alpha_max` 决定中心位置的 `c1_grid`
- RL 的 action space 本质上也是这组 `c1_grid`

也就是说，`alpha_max` 的口径如果偏了，后面的 oracle 和 RL action space 也会一起偏。

#### 3.4 CPP/CPS 和帧扩展长度

`alpha_max` 还会进入保护长度设计：

- `Lcpp = max(1, ceil(ell_max / (1 - alpha_max)))`
- `Lcps = max(1, ceil(alpha_max * N / (1 + alpha_max)))`

对应位置见：

- [`matlab/simulation/generate_dataset_timescaling_n256.m`](../matlab/simulation/generate_dataset_timescaling_n256.m)
- [`matlab/detectors/export_oracle_dataset_for_policy.m`](../matlab/detectors/export_oracle_dataset_for_policy.m)
- [`matlab/detectors/run_online_policy_oamp_oampnet.m`](../matlab/detectors/run_online_policy_oamp_oampnet.m)

### 4. 时间轴是什么粒度

当前代码的时间粒度可以概括成一句话：

- `sequence` 级初始化
- `frame` 级递推
- `path` 级参数向量
- `sample` 级 time-scaling 作用

展开说就是：

1. `sequence` 开始时初始化一次 `ell / alpha / h`
2. 每到新 `frame`，调用 `step_timevary_channel_state(...)`
3. 一个 `frame` 内，`ch_t.alpha` 是按路径定义的向量，不在 frame 内再更新
4. 但 `alpha(i)` 会通过 `idx=(1+alpha(i))n-ell(i)+Lcpp` 对每个 sample 生效，所以帧内不是“静态时延卷积”

额外还有两点：

- 检测器训练数据和 oracle/RL 数据导出里，SNR 是按 `sequence` 固定的，不是每帧乱跳
- `iid` 模式下没有时间递推，只有 `timevary_sequence` 才有真正的时变序列

### 5. 是否做了 resampling / residual Doppler 的设定

这部分最容易和文献口径混掉，当前代码需要明确区分：

#### 5.1 代码里“有 residual”，但不是“resampling 后 residual”的两级模型

当前 `common_with_path_residual` 里的 `residual` 指的是：

- 各路径相对公共运动项 `alpha_com_t` 的残差 `delta_alpha_t`

它表示的是：

- 公共运动之外的路径差异；

它**不是**显式写成：

- 粗补偿前 `alpha_raw`
- resampling 后 `alpha_residual`

这种两级 UWA 接收机模型。

#### 5.2 当前代码没有单独建模“粗补偿前 vs 粗补偿后”的 `alpha_max`

也就是说，当前代码里没有单独的：

- “原始大 time-scaling”
- “重采样补偿后的剩余小 time-scaling”

这两套变量。

当前实际情况更像是：

- 直接假设系统工作在一个较小的残余 `alpha_max` 上；
- 然后这同一个 `alpha_max` 同时用于：
  - 信道生成
  - `c1` 计算
  - `c1_grid` / oracle / RL action space
  - `Lcpp` / `Lcps`

#### 5.3 当前口径更像“直接建模 residual-scale alpha”

从数值上看，当前主链用的是：

- `alpha_max = 1e-4`

这更像“已经比较小的 residual time-scaling 量级”，而不是“补偿前的大尺度 UWA 原始多普勒”。

所以如果后续要和文献的 resampling 口径严格对齐，最需要补的一步不是改 `c1`，而是先把代码里拆成两层：

1. 粗 time-scaling / resampling 前的 `alpha_raw`
2. resampling 后进入 AFDM/DAF 设计的 `alpha_res`

## B. 这套代码更接近哪一种建模

如果直接回答“它到底属于哪一派”，当前代码更接近：

- **宽带按路径 time-scaling 建模**

而不是：

- **窄带统一 Doppler shift 建模**

原因是：

- 路径级参数是 `alpha(i)`，不是一个全局共享 Doppler 频移
- 离散时域信道写成了 `idx=(1+alpha(i))n-ell(i)+Lcpp`
- 有显式的时间缩放/采样位置变化，不只是频域相位旋转
- `c1`、`CPP/CPS`、`action grid` 又都围绕这个 `alpha_max` 设计

但也要看到它和文献的差别：

- 代码里是“逐 frame 更新一次路径参数”的离散实现
- 没有把连续时间的 `tau_i(t)=tau_i-\alpha_i t` 和 `nu_i(f)=nu_i+\alpha_i f` 原样显式写成解析式
- 也没有显式分离“resampling 前”和“resampling 后 residual”两层

所以最准确的表述是：

- **当前实现属于宽带 time-scaling 思路下的离散化、逐帧慢变版本**

## C. `alpha_max` 的代码口径

为了后面统一 AFDM_UWA_Channel、OAMPNet 和 RL 的设定，可以把当前代码里的 `alpha_max` 理解成：

- 一个直接进入 AFDM 设计与检测流程的“工作口径”

它当前承担了 4 个角色：

1. 约束每条路径的 `alpha_t` 范围
2. 决定 `c1_base`
3. 决定 `c1_grid` 的中心位置
4. 决定 `Lcpp/Lcps`

因此它目前不是“纯信道统计参数”，而是“信道建模 + 系统设计 + action 定义”的共同锚点。

如果后面要和 UWA 文献完全对齐，建议把这个单一口径拆成两类：

- `alpha_max_raw`
  - 补偿前、物理原始 time-scaling 上界
- `alpha_max_res`
  - resampling 后、真正进入 `c1` / `CPP/CPS` / RL grid 设计的剩余上界

## 2. 总体结构

### 2.1 两种数据模式

数据生成和策略数据导出都支持两种模式：

- `iid`
  - 每个样本独立初始化一次信道。
  - 没有真正的序列递推。
- `timevary_sequence`
  - 一条序列对应一个内部信道状态。
  - 第 1 帧调用 `init_timevary_channel_state(...)`
  - 后续帧调用 `step_timevary_channel_state(...)`

这套入口在下面两个文件中都能看到：

- [`matlab/simulation/generate_dataset_timescaling_n256.m`](../matlab/simulation/generate_dataset_timescaling_n256.m)
- [`matlab/detectors/export_oracle_dataset_for_policy.m`](../matlab/detectors/export_oracle_dataset_for_policy.m)

其中 `generate_dataset_timescaling_n256.m` 明确把 `mode='timevary_sequence'` 定义为“新的 sequence-level 递推信道”，并在 `use_timevary` 分支下按 `sequence_id / time_index` 推进信道状态。

## 3. 统一默认参数在哪里定义

默认超参数由 [`matlab/common/get_timevary_defaults.m`](../matlab/common/get_timevary_defaults.m) 统一给出。当前默认值的核心部分如下：

| 参数 | 默认值 | 含义 |
| --- | --- | --- |
| `rho_alpha` | `0.98` | 旧版独立路径 AR(1) 多普勒相关系数 |
| `rho_h` | `0.98` | 路径复增益 `h_t` 的 AR(1) 相关系数 |
| `alpha_max` | `1e-4` | 多普勒/时间尺度参数上限 |
| `doppler_mode` | `independent_path_ar1` | 默认多普勒模型 |
| `rho_acc` | `0.95` | 公共运动速度状态的递推相关系数 |
| `sigma_acc` | `0.03` | 公共运动扰动强度 |
| `rho_delta` | `0.90` | 路径残差多普勒的递推相关系数 |
| `sigma_delta` | `0.05` | 路径残差多普勒扰动强度 |
| `delta_max` | `0.15 * alpha_max` | 残差多普勒裁剪范围 |
| `motion_profile` | `smooth_ar` | 公共运动轨迹模式 |
| `path_projection_mode` | `ones` | 公共运动投影到各条路径的方式 |
| `num_frames` | `10` | 序列长度默认值 |
| `ell_mode` | `static` | 路径时延是否在序列内变化 |
| `pdp_mode` | `exp_fixed_per_sequence` | 每条序列内固定的指数型 PDP |

需要注意两点：

- `get_timevary_defaults.m` 的“全局默认”仍然是 `independent_path_ar1`。
- 但很多 paper 入口脚本会再把它覆盖成 `common_with_path_residual`，所以“当前 paper 主链路实际使用的默认”与“函数裸默认值”不是一回事。

## 4. 信道状态里到底包含什么

`init_timevary_channel_state.m` 在每条序列开始时一次性生成并保存以下状态：

- `ell`
  - 从 `[0, ell_max]` 随机采样 `P` 条路径时延。
  - 排序后再减去最小值，使最小时延归零。
- `pdp_power`
  - 按 `exp(-ell / ell_rms)` 生成指数型路径功率分布。
  - 当前实现中，`pdp` 在一条序列内固定不变。
- `h_t`
  - 初始复高斯路径增益，按 `sqrt(pdp_power)` 加权后再归一化。
- `alpha_t`
  - 路径级时间尺度/多普勒参数。
- `alpha_com_t`
  - 公共多普勒分量。
- `delta_alpha_t`
  - 相对公共分量的路径残差多普勒。
- `v_norm_t`
  - 归一化公共运动速度状态。
- `acc_t`
  - 归一化公共运动加速度状态。
- `beta_path`
  - 公共运动投影到各路径时使用的比例系数。

## 5. 每一帧如何递推

`step_timevary_channel_state.m` 负责序列内逐帧更新。当前代码里，时变主要体现在 `alpha_t`、`h_t`，以及可选的 `ell` 慢漂移。

### 5.1 多普勒模式 `doppler_mode`

当前支持 3 种模式。

#### 模式 A: `independent_path_ar1`

这是旧版基线。

- 每条路径的 `alpha_t` 独立做 AR(1)：
  - `alpha_t = rho_alpha * alpha_{t-1} + sqrt(1-rho_alpha^2) * alpha_max * randn`
- 然后裁剪到 `[-alpha_max, alpha_max]`
- 公共分量只是事后计算：
  - `alpha_com_t = mean(alpha_t)`
  - `delta_alpha_t = alpha_t - alpha_com_t`

这个模式保留着，但不是当前 `vdop` / `vdop_ctrl` 主链默认模式。

#### 模式 B: `common_only`

- 先递推一个公共速度状态 `v_norm_t`
- 再令：
  - `alpha_com_t = alpha_max * v_norm_t`
  - `alpha_t = beta_path .* alpha_com_t`
- 不再单独添加路径残差：
  - `delta_alpha_t = 0`

#### 模式 C: `common_with_path_residual`

这是当前 paper 相关时变信道最常用的模式。

- 先递推公共速度状态 `v_norm_t`
- 再生成公共多普勒：
  - `alpha_com_t = alpha_max * v_norm_t`
- 再对路径残差做 AR(1)：
  - `delta_alpha_t = rho_delta * delta_alpha_{t-1} + sigma_delta * alpha_max * randn`
- 残差会被裁剪到 `[-delta_max, delta_max]`
- 最终每条路径的多普勒为：
  - `alpha_t = beta_path .* alpha_com_t + delta_alpha_t`
- 最后再整体裁剪到 `[-alpha_max, alpha_max]`

这意味着：

- 大趋势由公共运动驱动；
- 路径之间仍保留少量不完全一致的 Doppler 残差；
- 比 `common_only` 更接近“同一平台运动 + 多路径细微差异”的建模方式。

### 5.2 公共运动是怎么来的

`v_norm_t` 不是直接随机跳变，而是由 `motion_profile` 决定。

当前支持 3 种 profile：

- `smooth_ar`
  - 旧版平滑 AR 递推。
- `approach_turn_recede`
  - 先接近、再转向、再远离，并叠加低频 heave 起伏。
- `maneuver_heave`
  - 更强的分段机动轨迹，并叠加 heave 振荡。

如果 `motion_profile='smooth_ar'`：

- `acc_t = rho_acc * acc_{t-1} + sigma_acc * randn`
- `v_norm_t = v_norm_{t-1} + acc_t`

如果使用 `approach_turn_recede` 或 `maneuver_heave`：

- 先按序列归一化时间 `tau = (frame_index - 1) / (num_frames - 1)` 计算目标轨迹 `target_v_norm_t`
- 再通过
  - `target_track_gain`
  - `target_blend`
  - `profile_jitter_std`
  控制当前速度对目标轨迹的跟踪强度和随机扰动

### 5.3 路径增益 `h_t` 的递推

`h_t` 每帧也会变化，规则是：

- `h_t = rho_h * h_{t-1} + sqrt(1-rho_h^2) * innovation`
- 再乘以固定的 `sqrt(pdp_power)`
- 最后整体归一化

这表示：

- 每条路径的复增益是相关衰落，不是每帧完全独立重抽；
- PDP 决定了各路径平均能量层级；
- 序列内总能量会被重新归一化，避免数值漂移。

### 5.4 路径时延 `ell` 的递推

当前默认是：

- `ell_mode = 'static'`

也就是：

- 一条序列内的路径时延不变；
- 时变主要来自 `alpha_t` 和 `h_t`。

代码也支持：

- `ell_mode = 'slow_drift'`

此时 `ell` 会以小概率、按小步长随机漂移，但当前 paper 入口没有启用这个选项。

## 6. 信道参数最终怎样进入物理信道矩阵

时变状态不会停留在抽象参数层，而是会被真正写进物理信道矩阵 `G`。

核心实现位于：

- [`matlab/simulation/generate_dataset_timescaling_n256.m`](../matlab/simulation/generate_dataset_timescaling_n256.m)
- [`matlab/detectors/run_online_policy_oamp_oampnet.m`](../matlab/detectors/run_online_policy_oamp_oampnet.m)

关键公式体现在 `build_timescaling_G_sparse(...)` 中：

- 采样位置：
  - `idx = (1 + alpha(i)) * n - ell(i) + Lcpp`
- 多普勒相位：
  - `phase = exp(1j * 2 * pi * (alpha(i) * fc) * (n * dt))`

含义可以直接理解成：

- `ell(i)` 控制第 `i` 条路径的时延偏移；
- `alpha(i)` 同时控制时间缩放和载波相位旋转；
- `h(i)` 控制该路径的复增益；
- 代码用线性插值把非整数采样位置写入稀疏矩阵 `G`；
- 后续再经 AFDM 解调得到等效信道矩阵 `Heff`。

所以从“代码层面”的角度说，时变信道最终是通过每帧不同的 `ch_t = {ell, alpha, h}` 生成不同的 `G` 和 `Heff`。

## 8. 序列内 SNR 是怎么处理的

当前代码不仅让信道按序列递推，SNR 也是按“序列粒度”处理的，而不是帧帧乱跳。

### 8.1 检测器训练数据生成

在 [`matlab/simulation/generate_dataset_timescaling_n256.m`](../matlab/simulation/generate_dataset_timescaling_n256.m) 中：

- 一条 `sequence` 刚初始化时会选一次 `seq_snr`
- 后续该序列内所有 `frame` 都复用这个 `seq_snr`

### 8.2 Oracle/RL 数据导出

在 [`matlab/detectors/export_oracle_dataset_for_policy.m`](../matlab/detectors/export_oracle_dataset_for_policy.m) 中：

- 每条 `sequence` 进入循环时随机选一个 `snr_pick_seq`
- 序列内各帧共用这个 SNR

### 8.3 在线评估

在 [`matlab/detectors/run_online_policy_oamp_oampnet.m`](../matlab/detectors/run_online_policy_oamp_oampnet.m) 中：

- 外层按 `snr_db_list` 循环；
- 当前 outer-loop 的 `snr_val` 会固定作用于这一轮 rollout 中的所有 sequence/frame。

因此，当前实验设定可以概括为：

- **信道随帧演化**
- **SNR 在单条序列内固定**

## 9. 时变信道信息是如何保存到数据集中的

为了让 Python 训练和评估能识别“这是时变序列数据”，MATLAB 导出的 `.mat` 数据里会额外保存这些元信息：

- `sequence_id`
- `time_index`
- `dataset_mode`
- `dataset_mode_code`
- `doppler_mode`
- `doppler_mode_code`
- `timevary_hparams`
- `alpha_com`
- `v_norm`
- `delta_alpha_rms`

这几项的意义是：

- `sequence_id/time_index`
  - 让后续代码知道哪些 frame 属于同一条序列。
- `dataset_mode/doppler_mode`
  - 让 Python 侧知道当前数据是不是 `timevary_sequence`，以及用的是哪种多普勒模式。
- `timevary_hparams`
  - 让 Python 能读到 MATLAB 侧实际使用的时变超参数。
- `alpha_com/v_norm/delta_alpha_rms`
  - 让后续分析或可选特征工程可以观测当前的公共运动与路径残差强度。

## 10. Python 侧如何读取这些时变信息

Python 侧由 [`python/rl_c1/env_c1_bandit.py`](../python/rl_c1/env_c1_bandit.py) 读取上述字段。

它会解析并标准化：

- `dataset_mode`
  - 识别 `iid` 或 `timevary_sequence`
- `doppler_mode`
  - 识别 `independent_path_ar1` / `common_only` / `common_with_path_residual`
- `timevary_hparams`
  - 从 `.mat` 的 `timevary_hparams` group 中抽取标量超参数

因此，MATLAB 里定义的时变信道并没有在导出到 Python 后丢失语义。

## 11. 是否把物理多普勒量直接送进策略特征

默认情况下，没有。

在 [`matlab/common/extract_online_state_features.m`](../matlab/common/extract_online_state_features.m) 中：

- `include_physical_doppler_state = false` 时
  - `alpha_com`
  - `v_norm`
  - `delta_alpha_rms`
  不会进入 policy state
- 只有显式开启 `include_physical_doppler_state = true` 时，这三个物理量才会被附加到 state feature

这说明当前主实验默认更偏向“在线可观测统计特征 + 历史上下文”，而不是直接把真实信道状态喂给策略。

## 12. 当前代码如何验证时变信道确实在演化

验证脚本：

- [`matlab/detectors/test_timevary_online_rollout.m`](../matlab/detectors/test_timevary_online_rollout.m)

它会检查：

- `time_index` 在同一序列内严格递增
- `h_t` 在相邻帧之间确实发生变化
- `alpha_com` 在相邻帧之间确实发生变化
- `delta_alpha_rms` 元数据存在

这说明当前代码并不是“名义上叫时变，实际每帧没变”。

## 13. 一句话总结

如果只保留一句话来概括当前实现，可以写成：

> 当前代码里的时变信道，是以 **sequence 为单位初始化**、以 **frame 为单位递推** 的宽带多径时间尺度信道；默认 paper 版本使用 **公共多普勒 + 路径残差** 的 `common_with_path_residual` 模型，并通过 `alpha_t / h_t / ell` 的逐帧更新生成不同的物理信道矩阵 `G` 和等效信道矩阵 `Heff`。
